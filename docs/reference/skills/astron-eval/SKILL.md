# AstronEval 路由技能

AstronEval 是评测相关场景的统一入口技能，职责不是自己完成所有评测工作，而是先识别用户到底要做哪一类评测，再把请求路由到最合适的子技能。凡是与”评测””评估””模型效果对比””任务提交与查看结果”相关的请求，优先从这里进入。

## 路由目标

当前 AstronEval 下有两类核心子技能。

`model-evaluation` 用于评测 AI 模型在真实业务任务上的输出质量。适合处理上传评测集、选择或配置评测维度、提交评测任务、查看评测进度、查看评测结果、继续评测、恢复评测这类请求。它关注的是“这个模型在业务任务上答得好不好”。

`skill-evaluation` 用于对本地 skill 进行自动化评测。支持四种评测场景：快速验证单个 skill 的效果、对比多个同类 skill、测试同一 skill 在不同驱动模型下的表现、以及 skill 版本对比（A/B Test）。适合处理”帮我评测这个 skill””跑一下 skill 评测””提交 skill 评测任务””哪个模型驱动这个 skill 效果更好””对比 v1 和 v2 版本的 skill”这类请求。它通过评测集（本地上传或平台自动合成）、评委模型打分，输出评测报告摘要和在线报告链接。

## 路由判断

收到请求后，先判断用户的目标属于下面哪一类。

第一类是“模型质量评测”。当用户想评测某个模型在问答、文案生成、信息分析、流程自动化、旅游出行、翻译等业务场景中的输出质量，或者已经有评测集、评测任务、评测结果需要继续处理时，路由到 `model-evaluation`。

第二类是”skill 评测”。当用户想对某个本地 skill 进行评测，或想比较不同 skill 版本的效果，或想知道不同驱动模型执行这个技能时哪个效果更好，路由到 `skill-evaluation`。

如果用户同时提到”评测”和”模型对比”，不要只看关键词，要看问题核心。只要问题核心是”比较不同驱动模型执行某个技能的效果”或”对比 skill 版本”，就应该优先路由到 `skill-evaluation`；只要问题核心是”评测某个模型在业务任务上的输出质量”，就应该优先路由到 `model-evaluation`。

## 模糊场景处理

当用户表达不够清楚时，不要直接假设，先用一句中文澄清核心目标。澄清重点只有一个：用户到底是想评测模型在业务任务上的效果，还是想比较不同 Claude 模型执行某个技能的表现。

可直接这样问：

“你现在是想做模型业务效果评测，还是想评测某个 skill 的效果（或对比不同版本、不同驱动模型）？”

只在确实无法判断时提问；能够根据上下文判断时，直接路由并说明理由。

## 不适用场景

以下情况不要路由到 AstronEval 的现有评测子技能：

- 代码评审、代码质量分析、PR review
- 系统接口压测、服务性能测试、数据库性能优化
- 模型训练、微调、调参、loss 分析
- 与评测无关的一般工程开发任务

这类请求虽然也可能出现”测试””评估””benchmark”等词，但不属于 AstronEval 当前的评测范围。

## 执行要求

一旦判断清楚意图，**立即静默调用对应子技能，不要输出任何路由说明**（如"你想评测 xxx，我来路由到 skill-evaluation"、"已选择子技能 xxx"等）。路由本身对用户不可见，子技能的第一个输出就是用户看到的第一句话。

对于 `model-evaluation`，关注模型评测场景、评测集、维度、任务状态和结果查看。

对于 `skill-evaluation`，关注目标技能、评测场景（快速验证 / 横评 / 驱动模型对比 / 版本对比）、评测集来源、驱动模型与评委模型配置，以及最终的评测报告。

## 扩展方式

后续 AstronEval 新增评测子技能时，在本文件中补充该技能解决的问题边界、典型触发语句和与现有技能的区分标准。新增的子技能仍应遵循同一原则：先判断评测目标，再路由到专门技能，而不是在入口技能中堆叠全部实现细节。
