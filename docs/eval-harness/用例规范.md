# 用例规范

## 用例结构

每个用例是一个 YAML 文件，定义完整的测试场景。用例通过 `instruction` 字段指定发送给 `claude -p` 的指令，通过 `fixtures` 字段控制 mock 脚本的返回值。

### 完整 Schema

```yaml
# 元信息
id: string                    # 唯一标识，格式：{stage}_{序号}_{简述}
name: string                  # 中文描述
stage: enum                   # init | build | set | execute
priority: enum                # P0（必须通过）| P1（应该通过）| P2（最好通过）
runtime: string               # 可选，默认 claude-code（可选 openclaw / hermes）
max_turns: int                # 可选，默认 100
timeout: int                  # 可选，默认 120（秒）

# 首轮指令（发送给 claude -p 的内容）
instruction: string           # 触发 Skill 的用户输入

# 环境 Fixture（控制 mock_eval_skill.py 的返回值）
# 两种格式：Object = 静态（每次返回相同）; Array = 序列（按调用顺序返回，末尾永久重复）
fixtures:
  check-token: object         # 静态：每次返回相同 JSON
  recent-tasks: object        # 静态：每次返回相同 JSON
  login: object               # 静态（可选）
  models: object              # 静态（可选）
  task-status:                # 序列（可选）：轮询场景按顺序返回
    - object                  #   第1次调用返回
    - object                  #   第2次调用返回
    - object                  #   第3次及之后永久返回（末尾重复）
  # ... 按 eval_skill.py 子命令名添加

# 多轮交互（可选，用 --resume 继续会话）
follow_ups:
  - user_input: string        # 后续轮次的用户输入
    fixtures_override: object # 可选，覆盖该轮的 fixture 返回值

# LLM Judge 上下文（可选，供软约束评分使用）
expected_behavior: string     # 简述期望行为，注入 Judge prompt 的 {expected_behaviors} 占位

# 断言
assertions:
  hard:                       # 硬约束（全部 PASS 才通过）
    rules: [string]           # 条件规则 ID 列表（仅 H7-H9 需声明）
                              # H0（Skill 加载）自动执行，失败 → SKIP
                              # H1-H6（通用规则）自动执行，无需声明
    custom:                   # 自定义断言
      - type: string
        pattern: string
        scope: string
  soft: string                # "all" 或指定维度列表
```

## 命名规范

### 文件命名

```
{stage}/{序号}_{场景简述}.yaml
```

示例：
- `init/01_fresh_no_history.yaml`
- `init/02_has_history_no_intent.yaml`
- `init/03_explicit_single_skill.yaml`
- `build/01_quick_verify_full_flow.yaml`

### ID 命名

```
{stage}_{序号}_{英文简述}
```

示例：
- `init_01_fresh_no_history`
- `init_02_has_history_no_intent`

## 用例示例

### 示例1：全新用户，无历史任务

```yaml
id: init_01_fresh_no_history
name: 全新用户首次触发，无历史任务
stage: init
priority: P0

instruction: "/skill-evaluation"

fixtures:
  check-token:
    status: valid
    expires_at: "2026-07-01T00:00:00Z"
  recent-tasks:
    tasks: []

expected_behavior: |
  无历史任务时，展示自然语言引导（含 xxx Skill 占位符），不展示历史任务表。

assertions:
  hard:
    rules: ["H9"]             # 仅声明条件规则；H0-H6 自动执行
    custom:
      - type: text_present
        pattern: "请描述你的评测需求"
        scope: first_text_output
      - type: text_absent
        pattern: "最近的评测任务"
        scope: all_text
  soft: all
```

  有历史任务时必须先展示历史任务表（Markdown 表格），同时提供"新建"入口。

        scope: first_text_output

## Fixture 设计

### 原则

Fixture 文件替代真实 API 响应，控制 Claude 执行环境的状态。mock_eval_skill.py 收到子命令后，从 `.eval-fixtures/{subcommand}.json` 读取对应 fixture 返回。

### Fixture 与子命令映射

| Fixture 键名 | 对应子命令 | 说明 |
|-------------|-----------|------|
| `check-token` | `eval_skill.py check-token` | 鉴权状态 |
| `login` | `eval_skill.py login` | 登录流程返回 |
| `recent-tasks` | `eval_skill.py recent-tasks` | 历史任务列表 |
| `models` | `eval_skill.py models` | 可用模型列表 |
| `search-skill` | `eval_skill.py search-skill` | Skill 搜索 |
| `convert-excel` | `eval_skill.py convert-excel` | Excel 格式转换 |
| `upload-dataset` | `eval_skill.py upload-dataset` | 上传评测集 |
| `datamaker` | `eval_skill.py datamaker` | 启动自动合成 |
| `datamaker-status` | `eval_skill.py datamaker-status` | 合成任务轮询 |
| `submit` | `eval_skill.py submit` | 提交评测任务 |
| `status` | `eval_skill.py status` | 评测任务状态轮询 |

### 序列 Fixture 示例

轮询类交互（如等待任务完成）需要按调用顺序返回不同响应：

```yaml
# execute 阶段用例：模拟任务从提交到完成
fixtures:
  check-token:
    status: valid
  submit:
    status: ok
    task_id: "eval-test-001"
  status:                   # Array = 序列模式
    - status: pending       # 第1次轮询
      progress: 0
    - status: running       # 第2次轮询
      progress: 60
    - status: completed     # 第3次及之后（末尾永久重复）
      progress: 100
      report_url: "https://example.com/report/001"
```

**规则**：Array 最后一个元素永久重复，模拟"最终态"。即使 Claude 多轮询几次，也能得到一致的终态响应。

### 最小 Fixture 集

每个阶段的用例至少需要以下 fixture：

| 阶段 | 必需 Fixture |
|------|-------------|
| init | `check-token` + `recent-tasks` |
| build | `check-token` + `models` + `search-skill` |
| set | `check-token` |
| execute | `check-token` + `submit` + `status`（序列） |

## 自定义断言类型

| type | 说明 | 参数 |
|------|------|------|
| `text_present` | 文本中必须包含 pattern | pattern, scope |
| `text_absent` | 文本中不能包含 pattern | pattern, scope |
| `regex_match` | 文本匹配正则 | pattern, scope |
| `regex_absent` | 文本不匹配正则 | pattern, scope |
| `format_check` | 格式检查 | check (markdown_table / numbered_list), scope |
| `tool_called` | 某工具被调用 | tool, count(可选) |
| `tool_not_called` | 某工具未被调用 | tool |
| `tool_param_check` | 工具调用参数检查 | tool, param_path, expected |
| `tool_order` | 工具调用顺序 | tools (有序列表) |

### scope 取值

| scope | 含义 |
|-------|------|
| `first_text_output` | Claude 首段文本输出 |
| `all_text` | 全部文本输出（跨所有轮次） |
| `turn_N` | 第 N 轮的文本输出（从 0 开始） |
| `last_text_output` | 最后一段文本输出 |

---

## 编写原则

1. **一个用例测一个分支**：不要在一个用例中覆盖过多分支，保持聚焦
2. **Fixture 最小化**：只配置当前场景需要的子命令返回值，未配置的子命令自动返回 `{"status": "ok"}`
3. **断言精准**：用 `text_present` 而非全文匹配，允许措辞变化
4. **优先级明确**：P0 覆盖核心路径和易错点，P1/P2 覆盖边缘场景
5. **可独立执行**：每个用例自包含，不依赖其他用例的执行结果
6. **规则声明最小化**：`assertions.hard.rules` 只声明条件规则（H7-H10）。H0（Skill 加载）和 H1-H6（通用规则）由框架自动执行，无需在用例中列出

## 多轮用例编写

多轮用例通过 `follow_ups` 字段定义后续轮次。执行引擎用 `--resume` 继续同一会话。

```yaml
# 多轮示例：首轮触发 → 用户选择场景 → 用户确认
instruction: "评测 my-skill"

fixtures:
  check-token: { status: valid }
  recent-tasks: { tasks: [] }

follow_ups:
  - user_input: "1"                    # 选择第一个场景
  - user_input: "确认"                  # 确认配置
    fixtures_override:                  # 第三轮需要新的 fixture
      models:
        models:
          - { id: "model-a", display_name: "Model A" }
          - { id: "model-b", display_name: "Model B" }
```

**注意**：`fixtures_override` 会在该轮执行前更新 `.eval-fixtures/` 目录中的对应文件，模拟环境状态变化。

## 用例积累规则

| 触发事件 | 动作 |
|----------|------|
| 发现行为 bug | 新增一条复现该 bug 的用例（防止回归） |
| 新增交互分支 | 新增覆盖该分支的用例 |
| 修改分支条件 | 更新受影响的已有用例 |
| 合并 Red Flag | 新增触发该 Red Flag 的用例（验证能检测到） |
