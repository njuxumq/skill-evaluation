# 设计方案

## 问题分析

Skill 迭代优化面临三个核心挑战：

| 挑战 | 说明 |
|------|------|
| **行为不可测** | Skill 是文档，不是代码——输出是 Claude 的运行时行为，无法用单元测试覆盖 |
| **约束多且隐性** | SKILL.md 中有 30+ 条 Red Flags、输出约束、术语约束，人工检查容易遗漏 |
| **分支路径多** | 5 种场景 × 多个触发条件 × 自动填充 vs 手动输入，组合爆炸 |

## 验证对象

Skill 控制的是 Claude 的运行时行为，拆解为 5 类可验证的行为：

| 行为类 | 含义 | 举例 |
|--------|------|------|
| **动作序列** | Claude 按什么顺序调用工具 | 第一个动作必须是 TodoWrite |
| **动作内容** | 工具调用的参数是否正确 | TodoWrite 的 content 是否为固定 4 阶段名 |
| **文本存在** | 输出中必须包含什么 | 有历史任务时必须展示任务表 |
| **文本缺失** | 输出中绝不能出现什么 | 禁用术语、过程描述、推理回放 |
| **条件分支** | 在特定条件下走哪条路径 | 模糊输入 → 5 选项表；明确意图 → 推断确认 |

## 执行方案：真实 CLI 驱动

### 设计理念

直接驱动真实的 Agent CLI（Claude Code / OpenClaw / Hermes），在真实运行环境中执行 Skill，收集完整 Transcript 后验证行为约束。

复用 [astroneval](https://github.com/user/astroneval) 的 Agent 框架模式（`InferenceService` + `BaseAgent`），将 Skill 行为测试建模为一种特殊的 inference 任务。

### 与 API Harness 方案的对比

| 维度 | API Harness（弃用） | 真实 CLI 驱动（采用） |
|------|---------------------|---------------------|
| Skill 加载 | 手动拼装 system prompt | **真实的 frontmatter 触发机制** |
| 工具行为 | Mock 返回 | **真实的 TodoWrite/Bash 调用** |
| eval_skill.py | 不执行 | **执行 mock 脚本，返回 fixture** |
| 保真度 | 中（模拟环境可能偏差） | **高（和用户真实使用一致）** |
| 基础设施 | 需从零实现 | **复用 astroneval Agent 框架** |
| 成本 | 低（Haiku API） | 中（真实 Claude 推理） |

## 核心组件

### 1. Workspace 构建器

为每个用例创建隔离的测试工作区：

```
test_workspace/
├── .claude/
│   └── skills/
│       └── skill-evaluation/       ← 拷贝当前版本 Skill 文档
│           ├── SKILL.md
│           ├── eval-init.md
│           ├── eval-build.md
│           ├── eval-set.md
│           ├── eval-execute.md
│           ├── processes/
│           ├── references/
│           └── eval_skill.py       ← 替换为 mock_eval_skill.py
├── .eval/                          ← 让 Skill 认为环境正常
│   └── auth.json                   ← fixture 写入
└── .eval-fixtures/                 ← 控制 mock 脚本的返回值
    ├── check-token.json
    ├── login.json
    ├── recent-tasks.json
    └── models.json
```

```python
class WorkspaceBuilder:
    """为测试用例创建隔离的工作环境"""

    def __init__(self, skill_source: Path):
        self.skill_source = skill_source  # 当前版本的 Skill 文档目录

    def build(self, case: TestCase, base_dir: Path, run_index: int = 0) -> Path:
        workspace = base_dir / f"{case.id}_run{run_index}"
        workspace.mkdir(parents=True, exist_ok=True)

        # 1. 拷贝 Skill 文档
        skill_dest = workspace / ".claude" / "skills" / "skill-evaluation"
        shutil.copytree(self.skill_source, skill_dest)

        # 2. 替换 eval_skill.py 为 mock 脚本
        mock_script = Path(__file__).parent / "mock_eval_skill.py"
        shutil.copy(mock_script, skill_dest / "eval_skill.py")

        # 3. 写入 fixture 文件
        fixtures_dir = workspace / ".eval-fixtures"
        fixtures_dir.mkdir(parents=True, exist_ok=True)
        for cmd, response in case.fixtures.items():
            (fixtures_dir / f"{cmd}.json").write_text(
                json.dumps(response, ensure_ascii=False)
            )

        # 4. 创建基础目录结构（让 Skill 不报错）
        (workspace / ".eval").mkdir(exist_ok=True)

        return workspace
```

### 2. Mock eval_skill.py

Claude 调用 Bash 执行 `eval_skill.py check-token ...` 时，实际运行的是这个 mock 脚本。它根据子命令名从 fixture 文件返回预设 JSON。

**支持两种 fixture 模式**：

| 模式 | fixture 内容 | 行为 | 适用场景 |
|------|-------------|------|----------|
| 静态 | JSON Object | 每次调用返回相同响应 | check-token、models 等无状态命令 |
| 序列 | JSON Array | 按调用顺序依次返回，末尾元素永久重复 | task-status 轮询等有状态交互 |

```python
#!/usr/bin/env python3
"""mock_eval_skill.py — 根据子命令从 fixture 文件返回预设响应。

支持两种 fixture 格式：
- Object（静态）：每次调用返回相同 JSON
- Array（序列）：按调用顺序依次返回，最后一个元素永久重复

Claude 调用 Bash 时执行的命令格式不变，返回的 JSON 格式不变，
只是数据来源从真实 API 变为本地 fixture 文件。
Claude 感知不到区别。
"""
import sys
import json
from pathlib import Path


def find_work_dir(args: list[str]) -> Path:
    """从命令行参数中提取 work-dir"""
    for i, arg in enumerate(args):
        if arg == "--work-dir" and i + 1 < len(args):
            return Path(args[i + 1])
        if arg == "--auth-file" and i + 1 < len(args):
            return Path(args[i + 1]).parent.parent
    return Path.cwd()


def main():
    args = sys.argv[1:]
    if not args:
        print(json.dumps({"status": "error", "message": "no subcommand"}))
        return

    subcommand = args[0]
    work_dir = find_work_dir(args)
    fixture_path = work_dir / ".eval-fixtures" / f"{subcommand}.json"

    if not fixture_path.exists():
        print(json.dumps({"status": "ok"}))
        return

    data = json.loads(fixture_path.read_text(encoding="utf-8"))

    if isinstance(data, list):
        # 序列模式：返回第一个元素，剩余写回（末尾元素永久保留重复）
        response = data[0]
        remaining = data[1:] if len(data) > 1 else data
        fixture_path.write_text(
            json.dumps(remaining, ensure_ascii=False), encoding="utf-8"
        )
        print(json.dumps(response, ensure_ascii=False))
    else:
        # 静态模式：每次返回相同响应
        print(json.dumps(data, ensure_ascii=False))


if __name__ == "__main__":
    main()
```

### 3. CLI Driver（执行引擎）

复用 astroneval 的 `ClaudeCodeAgent` 模式，驱动真实 CLI：

```python
class SkillTestRunner:
    """驱动 Agent CLI 执行 Skill 测试用例，收集 transcript"""

    def __init__(self, skill_source: Path, runtime: str = "claude-code"):
        self.workspace_builder = WorkspaceBuilder(skill_source)
        self.runtime = runtime

    async def run_case(self, case: TestCase, run_index: int = 0) -> TestResult:
        # 1. 创建隔离 workspace（每次重复使用独立目录，避免 fixture 污染）
        workspace = self.workspace_builder.build(case, self.tmp_dir, run_index)

        # 2. 获取 Agent 实例
        agent = get_agent(
            self.runtime,
            workspace,
            model_config=ModelDef(model=case.model or "claude-haiku-4-5"),
            max_turns=case.max_turns or 100,
            timeout=case.timeout or 120,
        )
        await agent.setup()

        # 3. 执行首轮指令
        result = await agent.run(case.instruction)

        # 4. 多轮交互（如有后续轮次）
        for follow_up in case.follow_ups:
            # 如有 fixtures_override，更新对应 fixture 文件
            if follow_up.fixtures_override:
                for cmd, response in follow_up.fixtures_override.items():
                    fixture_path = workspace / ".eval-fixtures" / f"{cmd}.json"
                    fixture_path.write_text(
                        json.dumps(response, ensure_ascii=False)
                    )

            result = await agent.run(
                follow_up.user_input,
                resume=result.session_id
            )

        # 5. 解析 transcript
        transcript = self.parse_transcript(result.transcript)

        # 6. 运行断言验证
        hard_result = self.validator.check(transcript, case)
        soft_result = await self.evaluator.score(transcript, case)

        return TestResult(
            case_id=case.id,
            hard=hard_result,
            soft=soft_result,
            transcript=transcript,
            raw_output=result.content,
            duration=result.duration_seconds,
            usage=result.usage,
        )
```

### 4. Transcript 解析器

Claude Code 的 transcript.jsonl 包含完整对话记录，可提取工具调用和文本输出：

```python
class TranscriptParser:
    """解析 Claude Code 的 transcript.jsonl，提取结构化事件"""

    def parse(self, transcript_path: str) -> ParsedTranscript:
        events = []
        with open(transcript_path) as f:
            for line in f:
                entry = json.loads(line)
                if entry.get("type") == "assistant":
                    for block in entry["message"]["content"]:
                        if block["type"] == "tool_use":
                            events.append(ToolCallEvent(
                                tool=block["name"],
                                input=block.get("input", {}),
                            ))
                        elif block["type"] == "text":
                            events.append(TextOutputEvent(
                                text=block["text"],
                            ))
        return ParsedTranscript(events=events)
```

`ParsedTranscript` 提供的查询接口：

```python
class ParsedTranscript:
    """结构化 transcript，供 Validator 查询。

    核心设计：Skill Scope（技能作用域）
    ─────────────────────────────────────
    Transcript 包含两个阶段的事件：
    1. Pre-Skill：Claude 决定调用 Skill 工具、加载文档等准备工作
    2. Post-Skill：Skill 内容注入后 Claude 的实际行为（断言验证的对象）

    所有行为断言（H1-H10）默认在 Post-Skill 作用域内执行。
    """

    def __init__(self, events: list[Event]):
        self.events = events
        self._skill_boundary = self._find_skill_boundary()

    def _find_skill_boundary(self) -> int:
        """定位 Skill 内容注入的边界。

        边界 = Skill tool_use 调用之后的下一个事件。
        如果 transcript 中没有 Skill 调用（不应发生），fallback 到起始位置。
        """
        for i, e in enumerate(self.events):
            if isinstance(e, ToolCallEvent) and e.tool == "Skill":
                return i + 1
        return 0  # Fallback: 无 Skill 调用时视为全部在作用域内

    @property
    def skill_loaded(self) -> bool:
        """Skill 是否被成功加载（transcript 中是否有 Skill 工具调用）"""
        return any(
            isinstance(e, ToolCallEvent) and e.tool == "Skill"
            for e in self.events
        )

    @property
    def skill_scope(self) -> list[Event]:
        """Skill 作用域内的事件（断言的默认操作对象）"""
        return self.events[self._skill_boundary:]

    @property
    def actions(self) -> list[ToolCallEvent]:
        """Skill 作用域内的工具调用"""
        return [e for e in self.skill_scope if isinstance(e, ToolCallEvent)]

    @property
    def text_outputs(self) -> list[TextOutputEvent]:
        """Skill 作用域内的文本输出"""
        return [e for e in self.skill_scope if isinstance(e, TextOutputEvent)]

    def first_text_output(self) -> str:
        texts = self.text_outputs
        return texts[0].text if texts else ""

    def all_text(self) -> str:
        return "\n".join(e.text for e in self.text_outputs)

    def first_tool_call(self, tool_name: str) -> ToolCallEvent | None:
        for e in self.actions:
            if e.tool == tool_name:
                return e
        return None

    def has_tool_call(self, tool_name: str) -> bool:
        return any(e.tool == tool_name for e in self.actions)

    def text_before_tool(self, tool_name: str) -> str:
        """返回 Skill 作用域内，第一次调用该工具之前的所有文本输出"""
        texts = []
        for e in self.skill_scope:
            if isinstance(e, ToolCallEvent) and e.tool == tool_name:
                break
            if isinstance(e, TextOutputEvent):
                texts.append(e.text)
        return "\n".join(texts)
```

### 5. Validator（规则验证器）— 两阶段裁决

验证分为两个阶段，对应 Skill 行为的两个前提条件：

```
Phase 0: Skill 是否被加载？
  ├── YES → 进入 Phase 1
  └── NO  → 裁决为 SKIP（触发可靠性问题，非 Skill 质量问题）

Phase 1: Skill 行为是否符合约束？
  ├── 全部 PASS → 裁决为 PASS
  └── 任一 FAIL → 裁决为 FAIL
```

```python
class TwoPhaseValidator:
    """两阶段裁决：先验证 Skill 加载，再验证行为约束"""

    def validate(self, transcript: ParsedTranscript, case: TestCase) -> Verdict:
        # Phase 0: Skill 加载检查
        if not transcript.skill_loaded:
            return Verdict.SKIP(
                reason="Skill 未被触发",
                signal="trigger_reliability"
            )

        # Phase 1: 行为约束检查（在 skill_scope 内执行）
        hard_result = self.hard_validator.check(transcript, case)
        return Verdict.PASS() if hard_result.all_pass() else Verdict.FAIL(hard_result)
```

**SKIP 的意义**：SKIP 不等于 PASS 也不等于 FAIL。它表明 Skill 的 `description` 字段对该输入的匹配力不够，需要优化触发文案。框架会额外统计触发率（trigger rate），作为 Skill 质量的独立指标。

硬约束规则详见 [断言体系.md](./断言体系.md)。

### 6. Evaluator（LLM 评分器）

对 ParsedTranscript 执行软约束评分，由独立 Claude 实例按 rubric 打分。

详见 [断言体系.md](./断言体系.md) 软约束部分。

### 7. Runner（编排器）

```python
class TestRunner:
    def run_all(self, version: str, repeat: int = 3) -> VersionResult:
        results = {}

        for case in self.load_cases():
            case_results = []
            for i in range(repeat):
                # 每次重复使用独立 workspace（避免 transcript 和序列 fixture 互相干扰）
                result = await self.skill_runner.run_case(case, run_index=i)
                case_results.append(result)

            results[case.id] = self.analyze_stability(case_results)

        return VersionResult(version=version, results=results)

    def analyze_stability(self, results: list[TestResult]) -> CaseVerdict:
        # Phase 0 统计：触发率
        skips = [r for r in results if r.verdict == Verdict.SKIP]
        effective = [r for r in results if r.verdict != Verdict.SKIP]

        if len(skips) == len(results):
            return CaseVerdict.SKIP(
                trigger_rate=0.0,
                reason="Skill 在所有执行中均未被触发"
            )

        trigger_rate = len(effective) / len(results)

        # Phase 1 统计：行为约束（仅对 effective 结果判定）
        hard_violations = [r for r in effective if r.verdict == Verdict.FAIL]
        if hard_violations:
            return CaseVerdict.FAIL(
                trigger_rate=trigger_rate,
                detail=f"{len(hard_violations)}/{len(effective)} 次违反硬约束"
            )

        # 软约束：计算平均分和方差
        scores = [r.soft.weighted_score for r in effective]
        avg = sum(scores) / len(scores)
        variance = sum((s - avg) ** 2 for s in scores) / len(scores)

        if variance > 1.0:
            return CaseVerdict.UNSTABLE(
                trigger_rate=trigger_rate,
                detail=f"行为不稳定（方差={variance:.2f}）"
            )

        return CaseVerdict.PASS(trigger_rate=trigger_rate, avg_score=avg)
```

**三态裁决汇总**：

| 裁决 | 含义 | 对迭代的指导 |
|------|------|-------------|
| PASS | Skill 加载 + 行为符合约束 | 可以合并 |
| FAIL | Skill 加载但行为违反约束 | 需修复 Skill 文档 |
| SKIP | Skill 未被触发 | 需优化 Skill description/trigger |
| UNSTABLE | 行为方差过大 | 需加强约束明确性 |

### 8. Comparator（版本对比器）

```python
class VersionComparator:
    def compare(self, baseline: VersionResult, candidate: VersionResult) -> Report:
        report = Report()

        for case_id in baseline.all_case_ids():
            old = baseline.get(case_id)
            new = candidate.get(case_id)

            # 行为回归：PASS→FAIL
            if old.verdict == PASS and new.verdict == FAIL:
                report.add_regression(case_id, old, new)
            # 触发回归：PASS→SKIP（Skill 不再被触发）
            elif old.verdict == PASS and new.verdict == SKIP:
                report.add_trigger_regression(case_id, old, new)
            # 改进
            elif old.verdict == FAIL and new.verdict == PASS:
                report.add_improvement(case_id, old, new)
            elif old.verdict == SKIP and new.verdict == PASS:
                report.add_improvement(case_id, old, new)

            # 触发率对比
            if old.trigger_rate is not None and new.trigger_rate is not None:
                report.add_trigger_delta(
                    case_id, new.trigger_rate - old.trigger_rate
                )

            # 软分对比（仅对有效结果）
            if new.avg_score is not None and old.avg_score is not None:
                report.add_score_delta(case_id, new.avg_score - old.avg_score)

        if report.regressions or report.trigger_regressions:
            report.verdict = "BLOCK - 存在回归，禁止合并"
        elif report.avg_score_delta < -0.2:
            report.verdict = "WARNING - 整体分数下降"
        else:
            report.verdict = "PASS - 无回归，可合并"

        return report
```

## 单轮 vs 多轮

### 单轮（claude -p）

单次 `claude -p` 调用覆盖 Skill 加载后的完整首轮行为：

```
用户输入 "评测 skill"
  → Claude 加载 Skill（真实 frontmatter 触发）
  → TodoWrite（进度表）
  → Bash（check-token）→ mock 返回 fixture
  → Bash（recent-tasks）→ mock 返回 fixture
  → 文本输出（选项表 / 历史任务表）
  → stop（等待用户输入）
```

这已覆盖 **阶段1 大部分验证需求**（动作序列、禁用术语、分支选择）。

### 多轮（--resume）

后续轮次用 `--resume` 继续同一会话：

```python
# Turn 1: 触发 Skill
result1 = await agent.run("评测 skill")
session_id = extract_session_id(result1.transcript)

# Turn 2: 用户选择"新建"
result2 = await agent.run("新建", resume=session_id)

# Turn 3: 用户描述需求
result3 = await agent.run("评测 my-skill", resume=session_id)
```

每个 turn 的 transcript 追加到同一 JSONL 文件，最终一起验证。

## 非确定性处理

Claude 的输出不是确定性的。同一输入可能产生不同措辞。

**策略**：

| 约束类型 | 判定规则 | 阈值 |
|----------|----------|------|
| 硬约束 | N 次执行全部通过 = PASS；任一违反 = FAIL | N=3 |
| 软约束 | 取 N 次平均分 | N=3 |
| 稳定性 | 方差 > 1.0 → 标记 UNSTABLE | 方差阈值=1.0 |

**UNSTABLE 的意义**：如果同一用例跑 3 次行为差异很大，说明 Skill 文档在这个场景的约束力不足——这本身就是需要优化的点。

## 多运行时支持

测试框架支持多种 Agent Runtime，验证 Skill 在不同框架下的行为一致性：

| Runtime | CLI 命令 | 说明 |
|---------|----------|------|
| claude-code | `claude -p` | 主要测试目标 |
| openclaw | `openclaw agent --local --message` | 验证跨框架一致性 |
| hermes | `hermes run` | 验证跨框架一致性 |

用例可指定 `runtime` 字段，默认为 `claude-code`。

## 模型分级策略

Skill 行为测试的信号质量取决于被测模型的指令遵循能力。模型太弱 → 大量 FAIL 归因为"模型能力不足"而非"Skill 约束不清晰"，产生无效噪声。

### 两级模型体系

| 级别 | 被测模型 | 目的 | 何时使用 |
|------|---------|------|----------|
| **Tier A（行为验证）** | Sonnet | 验证 Skill 约束是否被正确遵循 | P0/P1 日常迭代 |
| **Tier B（鲁棒性验证）** | Haiku | 验证 Skill 约束对弱模型是否足够明确 | P3 可选 |

**规则**：

- P0-P2 阶段**只使用 Tier A**，确保信号纯净
- Tier A 的 FAIL = Skill 文档约束力不足（可行动）
- Tier B 的 FAIL = Skill 文档对弱模型不够明确（优化方向，非阻塞）
- Judge 模型始终用 **Sonnet**

### 成本估算

| 项目 | Tier A (Sonnet) | Tier B (Haiku) |
|------|----------------|----------------|
| 每个用例单次执行 | ~8K input + ~2K output | ~8K input + ~2K output |
| 执行 10 用例 × 3 次 | ~300K tokens | ~300K tokens |
| LLM Judge 评分 | ~30K tokens (Sonnet) | ~30K tokens (Sonnet) |
| 一次完整验证总成本 | **~¥15-25** | **< ¥5** |
| 单个用例执行时间 | 15-45 秒 | 10-30 秒 |
| 完整验证时间 | 8-15 分钟（可并行） | 5-10 分钟（可并行） |

**成本控制策略**：

- 静态校验零成本，先过静态再跑 CLI（大部分低级错误在此拦截）
- 快速验证时只跑受影响用例（`--cases init/02`），不全量
- 并行执行不同用例的 workspace（独立隔离，无冲突）
- Tier B 仅在发布前可选执行
