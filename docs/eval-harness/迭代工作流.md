# 迭代工作流

## 概述

每次修改 Skill 文档后，按以下流程验证迭代是否正向。

---

## 验证流程

```
开发者修改 Skill 文档
        │
        ▼
┌──────────────────┐
│ 1. 静态校验       │  ← 秒级，零成本
│    lint_skill.py  │
└────────┬─────────┘
         │ PASS
         ▼
┌──────────────────┐
│ 2. 运行用例集     │  ← 分钟级，API 费用
│    run.py        │
│    (N=3 重复)    │
└────────┬─────────┘
         │ 全部 PASS
         ▼
┌──────────────────┐
│ 3. 版本对比       │  ← 自动执行
│    compare.py    │
│    (vs baseline) │
└────────┬─────────┘
         │ 无回归 + 分数不降
         ▼
┌──────────────────┐
│ ✅ 确认为正向迭代 │
│    更新 baseline │
└──────────────────┘
```

---

## 各环节详细

### Step 1: 静态校验

```bash
python eval/lint_skill.py
```

**检查内容**：

| 检查项 | 规则 |
|--------|------|
| 术语合规 | 扫描所有 Skill 文档，确保禁用术语不出现在非 references 文件中 |
| 场景名称一致 | scene-detection.md 中的选项文案与 SKILL.md、评测场景说明.md 完全一致 |
| activeForm 对照表完整 | 每个阶段文档中出现的步骤都有对应的 activeForm 映射 |
| 命令格式校验 | 阶段文档中引用的子命令在「脚本定义.md」中有定义 |
| TodoWrite content 固定 | 所有 TodoWrite 示例的 content 只包含 4 个固定阶段名称 |
| 文档引用完整 | processes/ 和 references/ 中被引用的文件都存在 |

**通过条件**：零 error。warning 可忽略但建议修复。

**失败处理**：修复后重跑，直到通过。

---

### Step 2: 运行行为用例

```bash
python eval/run.py --version v2.1 --repeat 3 --model sonnet
```

**模型配置**：`--model <name>` 指定 `eval/models/<name>.json` 中定义的模型。不指定时使用用户全局 claude 配置。日常迭代使用主力模型，鲁棒性验证可切换弱模型。

**执行逻辑**：

1. 加载 `eval/cases/` 下所有 YAML 用例
2. 对每个用例（每次重复使用独立 workspace，避免 fixture 和 transcript 污染）：
   - 构建隔离 Workspace（拷贝 Skill 文档 + 写入 fixture + 替换 mock 脚本）
   - 驱动 Agent CLI 执行 `claude -p "{instruction}"`（真实 Skill 加载）
   - 如有 `follow_ups`，用 `--resume` 继续会话
   - 收集 transcript（从 `~/.claude/projects/` 提取 JSONL）
   - **Phase 0**：检查 Skill 是否被加载（H0），未加载 → SKIP
   - **Phase 1**：解析 transcript 的 skill_scope，执行硬约束验证（H1-H10）
   - 执行 LLM 软约束评分
3. 每个用例重复 N 次（默认 3）
4. 聚合结果：计算 trigger rate、hard pass rate、soft score 均值和方差

**三态裁决**：

| 裁决 | 含义 | 后续动作 |
|------|------|----------|
| PASS | Skill 加载 + 行为符合约束 | 可合并 |
| FAIL | Skill 加载但行为违反约束 | 修复 Skill 文档 |
| SKIP | Skill 未被触发 | 优化 Skill description |
| UNSTABLE | 行为方差过大 | 加强约束明确性 |

**通过条件**：

| 条件 | 要求 |
|------|------|
| P0 用例硬约束 | 3/3 全部 PASS（SKIP 不算 PASS） |
| P1 用例硬约束 | 至少 2/3 PASS |
| 触发率 | >= 90%（所有用例的 Skill 加载成功率） |
| 稳定性 | 无 UNSTABLE 标记 |
| 软分均值 | >= 3.5（满分 5） |

**失败处理**：

1. 查看失败用例的 transcript（存储在 `eval/results/{version}/`）
2. 定位是 Skill 文档问题还是用例过严
3. 修改 Skill 文档 → 重跑 Step 1 + Step 2

---

### Step 3: 版本对比

```bash
python eval/compare.py --baseline v2.0 --candidate v2.1
```

**执行逻辑**：

1. 加载 baseline 和 candidate 的结果
2. 逐用例对比 verdict 变化
3. 计算软分 delta
4. 生成对比报告

**通过条件**：

| 条件 | 要求 |
|------|------|
| 回归 | 零回归（任何 PASS->FAIL 都算回归） |
| 退化 | 零退化（PASS->UNSTABLE） |
| 软分 delta | 平均不低于 -0.1 |

**报告输出示例**：

```
══════════════════════════════════════════════════════════════
  Skill 迭代验证报告：v2.0 -> v2.1
══════════════════════════════════════════════════════════════

  用例总数：12 | 通过：10 | 回归：1 | 改进：1

  回归（必须修复）：
  ────────────────────────────────────────────────────────
  [init/02_has_history_no_intent]
    v2.0: PASS (3/3)
    v2.1: FAIL (1/3 违反 "历史任务必须优先展示")
    原因分析：修改了首个输出规则的措辞导致 Claude 误判

  改进：
  ────────────────────────────────────────────────────────
  [build/03_auto_fill_from_context]
    v2.0: FAIL (输出了"已识别 Skill 名称为 xxx")
    v2.1: PASS (静默填充，无多余输出)

  软分变化（平均）：
    简洁度：  4.2 -> 4.5 (+0.3)
    正确性：  4.0 -> 3.7 (-0.3)  <- 注意
    自然度：  4.1 -> 4.2 (+0.1)
    体验：    3.9 -> 4.0 (+0.1)

  ────────────────────────────────────────────────────────
  结论：BLOCK - 修复回归后重新验证
══════════════════════════════════════════════════════════════
```

**失败处理**：

1. 回归 → 必须修复 Skill 文档，消除回归后重新验证
2. 软分下降 → 分析具体维度，判断是否可接受的 trade-off
3. 全部通过 → 更新 baseline 版本

## Baseline 管理

### 更新 baseline

当一次迭代通过全部验证后：

```bash
python eval/run.py --version v2.1 --set-baseline
```

这会将 `eval/results/v2.1/` 标记为新的 baseline，后续对比以此为基准。

### 版本历史

```
eval/results/
├── v1.0/               # 初始版本
│   ├── run_20260601_143022.json
│   └── summary.json
├── v2.0/               # 第一次优化
│   ├── run_20260605_091500.json
│   └── summary.json
├── v2.1/               # 当前版本
│   ├── run_20260609_103000.json
│   └── summary.json
└── baseline -> v2.0/   # 当前 baseline（符号链接）
```

## 不同修改类型的验证策略

| 修改类型 | 最低验证要求 |
|----------|-------------|
| 优化交互措辞 | Step 1 + Step 2（受影响的用例） |
| 新增评测场景 | Step 1 + Step 2（全部用例）+ 新增场景用例 |
| 修改分支条件 | Step 1 + Step 2 + Step 3（必须对比） |
| 新增子命令 | Step 1 + 新增涉及该命令的用例 |
| 重大重构 | Step 1 + Step 2 + Step 3（全量对比） |

## 快速验证 vs 完整验证

### 快速验证（日常开发）

```bash
# 只跑静态 + 受影响的用例（单次）
python eval/lint_skill.py
python eval/run.py --version dev --cases init/02 --repeat 1
```

适用：小幅措辞调整、格式修正。

### 完整验证（准备发布）

```bash
# 全量用例 x 3 次 + 版本对比
python eval/lint_skill.py
python eval/run.py --version v2.1 --repeat 3 --model sonnet
python eval/compare.py --baseline v2.0 --candidate v2.1
```

适用：修改分支逻辑、新增场景、重大重构。

### 鲁棒性验证（可选，P3）

```bash
# 用弱模型验证 Skill 约束是否足够明确
python eval/run.py --version v2.1 --repeat 3 --model haiku
```

适用：验证 Skill 对指令遵循能力较弱的模型是否仍有足够约束力。弱模型的 FAIL 不阻塞合并，但可作为优化方向。

## 常见问题

### Q: 用例失败了，是 Skill 问题还是用例问题？

**判断方法**：

1. 查看 transcript，Claude 的行为是否合理
2. 如果 Claude 行为合理但用例判 FAIL → 用例断言过严，调整用例
3. 如果 Claude 行为不合理 → Skill 文档约束力不足，修改文档

### Q: UNSTABLE 怎么修？

**原因**：Skill 文档在该场景的约束不够明确，Claude 每次理解不一致。

**修复方法**：
- 在 SKILL.md 或阶段文档中加强该场景的约束描述
- 增加 Red Flag 条目明确禁止错误行为
- 增加示例让 Claude 理解期望输出

### Q: 软分持续偏低怎么办？

**诊断**：
- 简洁度低 → 检查是否有冗余的引导句或过渡句
- 正确性低 → 检查分支条件描述是否清晰
- 自然度低 → 检查中文措辞是否僵硬
- 体验低 → 检查选项表是否清晰，信息层次是否合理
